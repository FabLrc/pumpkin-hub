Architecture

Architecture technique de la plateforme Pumpkin Hub.

Vue d'ensemble

Pumpkin Hub suit une architecture monorepo multi-services orchestrée par Docker Compose. Le frontend communique avec l'API REST, qui gère la logique métier et interagit avec les couches de données.

Tous les services partagent le réseau Docker pumpkin-network et communiquent via leurs noms de service (ex: postgres:5432).

API Rust (Axum)

Structure du code

api/src/
├── main.rs          # Point d'entrée, configuration serveur
├── lib.rs           # build_app() — construction de l'application
├── config.rs        # Configuration typée (env vars)
├── state.rs         # AppState partagé entre handlers
├── error.rs         # AppError — gestion d'erreurs centralisée
└── routes/
    ├── mod.rs       # Agrégation des routes v1
    └── health.rs    # Endpoint /api/v1/health

Pattern build_app()

L'application est construite via une fonction build_app() dans lib.rs qui retourne un Router. Ce pattern permet de tester l'application complète sans démarrer de serveur HTTP. 

// lib.rs (simplifié)
pub fn build_app(state: AppState) -> Router {
    Router::new()
        .nest("/api/v1", v1_routes())
        .with_state(state)
        .layer(middleware_stack())
}

Frontend (Next.js)

Structure

frontend/
├── app/
│   ├── layout.tsx    # Layout racine
│   ├── page.tsx      # Page d'accueil
│   └── globals.css   # Styles globaux (Tailwind)
├── public/           # Assets statiques
├── next.config.ts    # Configuration Next.js
└── tsconfig.json     # Configuration TypeScript

Le frontend utilise l'App Router de Next.js 16 avec React 19 et Tailwind CSS 4. Le design suit la direction artistique "Brutaliste Industrielle".

Couche données

PostgreSQL

  • Base de données relationnelle principale
  • Accès via SQLx (compile-time checked queries)
  • Migrations gérées par SQLx CLI
  • Volume Docker persisté pour les données

Meilisearch

  • Moteur de recherche full-text haute performance
  • Indexation des plugins pour la recherche instantanée
  • Support natif de la typo-tolerance et des filtres facettés
  • Volume Docker persisté pour les index

Middleware Stack

L'API utilise un stack de middlewares Tower ordonnés pour le traitement des requêtes :

OrdreMiddlewareRôle
1TraceLayerLogging structuré des requêtes/réponses
2CorsLayerGestion CORS (permissive en dev)
3SetRequestIdLayerID unique par requête (traçabilité)
4TimeoutLayerTimeout de 30s par requête
5CompressionLayerCompression gzip des réponses
