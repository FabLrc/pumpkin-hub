Contribuer

Guide de contribution au projet Pumpkin Hub.

Bienvenue
Toute contribution est la bienvenue, qu'il s'agisse d'un bug fix, d'une nouvelle feature, d'une amélioration de la documentation ou d'un report de bug. Merci de prendre le temps de lire ce guide avant de soumettre votre contribution.

Workflow Git

  1. Fork le dépôt sur GitHub
  2. Clone votre fork en local
  3. Créer une branche depuis develop (jamais depuis master)
  4. Implémenter les changements avec des commits atomiques
  5. Pousser la branche et ouvrir une Pull Request vers develop
# Exemple de workflow
git clone https://github.com/VOTRE-USERNAME/pumpkin-hub.git
cd pumpkin-hub
git checkout -b feature/ma-feature develop

# ... développement ...

git push origin feature/ma-feature

Branches

BrancheRôle
masterProduction stable, releases taggées
developIntégration, base des PRs
feature/*Nouvelles fonctionnalités
fix/*Corrections de bugs
docs/*Améliorations de la documentation

Conventions de code

Rust (API)

  • Formatage avec cargo fmt (non négociable)
  • Analyse statique : cargo clippy -- -D warnings sans warnings
  • Types HTTP via axum::http, pas la crate http directement
  • Nouvelles routes dans routes/{feature}.rs, mergées dans v1_routes()
  • Gestion d'erreurs via AppError — jamais de unwrap() dans le code de production

TypeScript (Frontend)

  • Linting avec ESLint (config Next.js) : npm run lint
  • Type-check strict : npx tsc --noEmit
  • Composants dans le répertoire app/ (App Router)
  • Styles via Tailwind CSS, respect du design system brutaliste

Principes généraux

  • Boy Scout Rule — Laisse le code plus propre que tu ne l'as trouvé
  • SOLID — Single Responsibility, interfaces minimales, dépendances inversées
  • KISS — La solution la plus simple est souvent la meilleure
  • DRY — Factorise sans sur-ingénierie
  • YAGNI — N'implémente pas de fonctionnalités "au cas où"

Messages de commit

Les messages de commit suivent la convention Conventional Commits : 

type(scope): description courte

[corps optionnel]

[footer optionnel]

Types autorisés

TypeDescription
featNouvelle fonctionnalité
fixCorrection de bug
docsDocumentation uniquement
styleFormatage, point-virgules (pas de changement de logique)
refactorRefactorisation sans changement de comportement
testAjout ou modification de tests
choreTâches de maintenance (CI, deps, config)

Scopes courants

api, frontend, docs, docker, ci, auth, plugins, search

Exemples

feat(api): add plugin CRUD endpoints
fix(frontend): correct search input focus state
docs: update roadmap with Phase 2
chore(ci): add Rust cache to workflow

Pull Requests

  • Le titre suit la convention de commit (feat(scope): ...)
  • Description claire du changement et de la motivation
  • Référence les issues liées (ex: Closes #42)
  • La CI doit passer (lint, clippy, tests, build)
  • Au moins 1 review approbative avant merge
Astuce
Pour les changements significatifs, ouvrir d'abord une issue pour discuter de l'approche avant de commencer l'implémentation. Cela évite le travail inutile et permet un feedback précoce.

Qualité du code

Avant de soumettre, vérifier que :

Backend

# Formatage
cargo fmt --all -- --check

# Analyse statique
cargo clippy -- -D warnings

# Tests
cargo test

Frontend

# Linting
npm run lint

# Type-checking
npx tsc --noEmit

# Build
npm run build
Important
La CI bloque le merge si l'un de ces checks échoue. Exécutez-les en local avant de pousser pour gagner du temps.

Ajouter une nouvelle feature (API)

Procédure pour ajouter un nouveau module de routes :

  1. Créer le fichier api/src/routes/{feature}.rs
  2. Implémenter les handlers avec les types AppState et AppError
  3. Déclarer le module dans routes/mod.rs (mod {feature};)
  4. Merger les routes dans v1_routes()
  5. Écrire les tests unitaires et/ou d'intégration
// api/src/routes/plugins.rs
use axum::{routing::get, Json, Router};
use crate::state::AppState;

pub fn routes() -> Router<AppState> {
    Router::new()
        .route("/plugins", get(list_plugins))
}

async fn list_plugins() -> Json<serde_json::Value> {
    // implementation
    todo!()
}
// api/src/routes/mod.rs
mod health;
mod plugins; // <-- ajouter ici

fn v1_routes() -> Router<AppState> {
    Router::new()
        .merge(health::routes())
        .merge(plugins::routes()) // <-- et ici
}
