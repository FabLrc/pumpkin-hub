Architecture

Organisation technique et principes architecturaux de Pumpkin Hub.

Vue d'ensemble

Pumpkin Hub suit une architecture client-serveur classique avec une séparation nette entre le frontend (Next.js) et le backend (API Rust/Axum). Les services communiquent via des appels REST JSON sous le préfixe /api/v1.

Backend — API Rust

Principes directeurs

  • Testabilité — build_app(config) dans lib.rs permet de construire l'app sans bind réseau, facilitant les tests d'intégration.
  • Separation of Concerns — Configuration, état, erreurs et routes sont dans des modules distincts.
  • Graceful Shutdown — Écoute CTRL+C et SIGTERM pour un arrêt propre des connexions.

Pile middleware (Tower)

Les middlewares sont appliqués dans l'ordre suivant via ServiceBuilder :

OrdreMiddlewareRôle
1SetRequestIdLayerAssigne un UUID unique (x-request-id) à chaque requête
2TraceLayerTraces structurées avec span par requête (tracing)
3PropagateRequestIdLayerPropage le request ID dans les headers de réponse
4TimeoutLayerTimeout global de 30s → 504 Gateway Timeout
5CompressionLayerCompression Gzip des réponses
6CorsLayerPolitique CORS multi-origines configurable

Gestion de configuration

La struct Config est chargée depuis les variables d'environnement via Config::from_env(). Un ConfigError typé est retourné pour les variables manquantes ou invalides. L'API échoue au démarrage (fail-fast) si la configuration est incorrecte.

Gestion d'erreurs

Le type AppError unifie toutes les erreurs applicatives et implémente IntoResponse d'Axum pour une conversion automatique en réponse HTTP JSON :

VarianteHTTP StatusComportement
NotFound404Ressource introuvable
UnprocessableEntity422Données invalides (message transmis au client)
Unauthorized401Accès non autorisé
Internal500Erreur interne (loguée côté serveur, jamais exposée)

Routage

Les routes sont organisées en modules par fonctionnalité et assemblées dans routes/mod.rs sous le préfixe /api/v1. Pour ajouter une nouvelle fonctionnalité, créer un fichier routes/{feature}.rs et le merger dans v1_routes().

Frontend — Next.js

Structure App Router

Le frontend utilise le App Router de Next.js 16 avec React 19. Les styles sont gérés par Tailwind CSS 4 avec le plugin PostCSS.

Design System — Brutaliste Industrielle

Le design system du projet suit ces principes :

  • Typographie — Raleway (titres et corps) + JetBrains Mono (code et données techniques)
  • Palette — Fond noir (#0a0a0a), texte neutre (#e5e5e5), accent orange (#f97316)
  • Géométrie — Zéro arrondi (border-radius: 0), bordures fines neutres (#262626)
  • Effets — Texture bruit (noise overlay), scanlines subtiles, glow orange sur focus

Infrastructure Docker

L'environnement de développement est orchestré par docker-compose.yml avec 4 services. Les données PostgreSQL et Meilisearch sont persistées via des volumes Docker. Le cache Cargo est également monté en volume pour accélérer les recompilations.

Attention
Le fichier docker-compose.yml est destiné au développement local uniquement. Les mots de passe et clés sont des valeurs par défaut qui ne doivent pas être utilisées en production.

CI/CD — GitHub Actions

Le workflow ci.yml s'exécute sur chaque push et pull request vers les branches master et develop. Deux jobs parallèles valident le frontend et le backend.

Frontend CI

  1. Lint — npm run lint (ESLint)
  2. Type-check — npx tsc --noEmit
  3. Build — npm run build

Backend CI

  1. Formatage — cargo fmt --all -- --check
  2. Analyse statique — cargo clippy -- -D warnings
  3. Tests — cargo test
Optimisations
Le cache NPM (package-lock.json) et le cache Rust (Swatinem/rust-cache) sont activés pour réduire les temps de build CI.
